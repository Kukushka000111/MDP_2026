## 1. Что это за проект одной фразой

**«Культурный Навигатор»** — это веб-приложение (сайт), где жители города видят **ленту мероприятий** на карте и в списке, могут **записаться**, добавить в **избранное**, оставить **отзыв**, а организаторы — **создавать события**. События от жителей проходят **модерацию** администратором перед публикацией.

Технически это **клиент–серверное приложение**: браузер (React) + API (Node.js/Express) + база PostgreSQL.

---

## 2. Зачем он нужен (предметная область)

| Проблема | Решение в проекте |
|----------|-------------------|
| События разбросаны по соцсетям и афишам | Единая лента с фильтрами и картой |
| Непонятно, есть ли ещё места | Лимит участников и счётчик записей |
| Нельзя доверять любой публикации | Премодерация событий типа «от жителей» |
| Нужны официальные и народные события | Типы `OFFICIAL` и `COMMUNITY` |
| Организатору нужны заявки | Запись со статусом PENDING → организатор одобряет/отклоняет |

**Целевой город в интерфейсе:** Кострома (центр карты задан в `client/src/constants.js`).

## 3. Общая архитектура

### 3.1. Три уровня (классическая схема)

```
┌─────────────┐      HTTP (JSON)       ┌─────────────┐      SQL        ┌──────────────┐
│   Браузер   │  ◄─────────────────► │   API       │  ◄───────────► │  PostgreSQL  │
│   React     │      /api/...         │   Express   │                │   (данные)   │
└─────────────┘                      └─────────────┘                └──────────────┘
```

- **Презентационный уровень** — React в браузере: кнопки, формы, карта, лента.
- **Бизнес-логика** — Express на Node.js: проверки, права, запросы к БД.
- **Уровень данных** — PostgreSQL: пользователи, события, записи, отзывы.

Фронтенд **не ходит в базу напрямую** — только через REST API.

### 3.2. В Docker (как на защите)

```
Браузер → localhost:8080 (nginx, контейнер web)
              ├── /           → статические файлы React (собранный Vite)
              └── /api/*      → прокси на контейнер api:4000
                                    └── контейнер db (PostgreSQL)
```

Файл `client/nginx.conf`: запросы на `/api/` перенаправляются на backend. Поэтому в production фронт обращается к API по относительному пути `/api`, без указания порта 4000.

### 3.3. Monorepo (один репозиторий — два пакета)

В корне `package.json` указаны **npm workspaces**: `server/` и `client/`. Одна команда `npm install` в корне ставит зависимости для обоих частей.

## 4. Технологии: что за что отвечает

### 4.1. Фронтенд (папка `client/`)

| Технология | Зачем в проекте |
|------------|-----------------|
| **React 18** | UI из компонентов; состояние (`useState`), эффекты (`useEffect`), один корневой `App.jsx` |
| **Vite 5** | Сборщик и dev-сервер: быстрый запуск `npm run dev`, production-сборка в `dist/` |
| **Tailwind CSS 3** | Стили через классы в разметке (`bg-indigo-600`, `rounded-2xl`); тёмная тема через класс `dark` на `<html>` |
| **Leaflet + react-leaflet** | Интерактивная карта OpenStreetMap, маркеры событий |
| **Hash-роутинг** | Навигация без React Router: `#feed`, `#event/uuid`, `#login` — меняется `window.location.hash` |

**Почему не Next.js?** Проект — SPA (Single Page Application): одна HTML-страница, переключение «экранов» внутри React. Для курсовой этого достаточно.

### 4.2. Бэкенд (папка `server/`)

| Технология | Зачем в проекте |
|------------|-----------------|
| **Node.js** | Среда выполнения JavaScript на сервере |
| **Express 4** | HTTP-сервер, маршруты (`router.get`, `router.post`), middleware |
| **pg** | Драйвер PostgreSQL, пул соединений (`server/src/db/pool.js`) |
| **jsonwebtoken (JWT)** | Токен после входа; клиент шлёт `Authorization: Bearer <token>` |
| **bcryptjs** | Хеш пароля при регистрации; при входе сравнение без хранения пароля в открытом виде |
| **Zod** | Валидация тела/query/params запросов в `server/src/schemas/` |
| **cors** | Разрешение запросов с другого origin в dev (клиент на :5173, API на :4000) |
| **dotenv** | Переменные окружения из `server/.env` |

### 4.3. База и инфраструктура

| Технология | Зачем |
|------------|--------|
| **PostgreSQL 16** | Реляционная БД, UUID, ENUM-типы, внешние ключи, индексы |
| **Docker Compose** | Три сервиса: `db`, `api`, `web` — поднять проект одной командой |

## 6. Как работает клиент (фронтенд)

### 6.1. Точка входа

1. `index.html` → `main.jsx` → компонент `App`.
2. `main.jsx` вызывает `initTheme()` — читает тему из `localStorage`, вешает класс `dark` на `<html>`.

### 6.2. Навигация без React Router

Используется **hash в URL**:

| Hash | Экран |
|------|--------|
| `#feed` | Главная: лента + карта |
| `#event/<id>` | Карточка мероприятия |
| `#login`, `#register` | Вход / регистрация |
| `#profile` | Профиль |
| `#attending`, `#my-events` | Вкладки «Моё» |
| `#favorites` | Избранное |
| `#create-event` | Форма события |
| `#moderation` | Панель админа |

Функция `navigate()` в `App.jsx` меняет `window.location.hash` и состояние `page`.

### 6.3. Где хранятся данные в браузере

| Ключ | Содержимое |
|------|------------|
| `localStorage: cultural-navigator:token` | JWT после входа |
| `localStorage: cultural-navigator:guest-favorites` | ID избранного для гостя |
| `localStorage: cultural-navigator:theme` | `light` / `dark` (дублируется в БД для авторизованных) |

После входа избранное гостя **сливается** на сервер (`addServerFavorite` для каждого id).

### 6.4. Как клиент ходит на сервер

Файл `client/src/api.js`:

- Базовый URL: `import.meta.env.VITE_API_URL` или `/api` в Docker.
- Защищённые запросы: заголовок `Authorization: Bearer ${token}`.
- Ошибки API разбираются в `parseApiError` и показываются через `Toast`.

### 6.5. Главные компоненты

| Компонент | Назначение |
|-----------|------------|
| `AppHeader` | Логотип, меню, «+ Создать», избранное, колокольчик, профиль, переключатель темы |
| `FeedSection` | Фильтры, список карточек, карта Leaflet |
| `EventFeedCard` | Карточка в ленте |
| `EventDetailSection` | Страница события: описание, запись, отзывы, контакты |
| `EventFormSection` | Создание/редактирование + `EventPointPicker` (клик по карте) |
| `MyActivitySection` | «Я иду» / «Я организую» |
| `ModerationPanel` | Очередь модерации + жалобы |
| `RegisterPage` / `LoginPage` | Регистрация с капчей и проверкой силы пароля |

Состояние сосредоточено в `App.jsx` (много `useState`) — для курсовой это нормально; альтернатива — Redux/Context, в проекте не используется.

## 7. Как работает сервер (бэкенд)

### 7.1. Запуск

`server/src/server.js` → слушает порт (4000) → подключает `app.js`.

`app.js` регистрирует префиксы:

```
/api/auth          → routes/auth.js
/api/events        → routes/events.js
/api/admin         → routes/admin.js (только ADMIN)
/api/meta          → categories, geocode
/api/favorites
/api/reviews
/api/me            → профиль, мои события, тема
/api/users         → публичный профиль
/api/notifications
```

Плюс `GET /health` — проверка, что API жив (для Docker и мониторинга).

### 7.2. Цепочка обработки запроса

```
HTTP запрос
  → cors, express.json()
  → маршрут (например POST /api/events)
  → requireAuth (если нужен) — проверка JWT
  → validate(schema) — Zod проверяет body/query/params
  → обработчик — SQL через pool.query()
  → JSON ответ или next(error)
  → error-handler — единый формат ошибок
```

### 7.3. Пул соединений с БД

`server/src/db/pool.js` — один пул `pg` на всё приложение. SQL пишется вручную (не ORM вроде Prisma). Параметры через `$1, $2` — защита от SQL-инъекций.

### 7.4. Валидация Zod

Пример: перед созданием события схема в `server/src/schemas/events.js` проверяет типы полей, даты, URL картинки. Если ошибка — **400** с текстом на русском.

Клиент дублирует часть проверок (`validateEventForm` в `utils.js`) для UX без ожидания ответа сервера.

## 8. База данных

### 8.1. Основные таблицы

| Таблица | Что хранит |
|---------|------------|
| `users` | Логин, email, `password_hash`, имя, профиль, роль, тема |
| `categories` | Категории («Кино & Театр», «Спорт & …») |
| `events` | Мероприятие: текст, адрес, координаты, тип, статус модерации, лимит мест |
| `event_registrations` | Заявка пользователя на событие + статус PENDING/APPROVED/REJECTED |
| `favorites` | Связь user ↔ event |
| `reviews` | Оценка 1–5 + текст, один отзыв на пару user+event |
| `notifications` | Уведомления пользователю (модерация, заявки и т.д.) |
| `event_reports` | Жалобы на опубликованные события |

### 8.2. Важные ENUM-типы (PostgreSQL)

- `user_role`: `USER`, `ADMIN`
- `event_type`: `OFFICIAL`, `COMMUNITY`
- `moderation_status`: `PENDING`, `NEEDS_EDIT`, `APPROVED`, `REJECTED`
- `registration_status`: `PENDING`, `APPROVED`, `REJECTED`

### 8.3. Логика модерации событий

При **создании** (`POST /api/events`):

- Если тип **OFFICIAL** и создатель **ADMIN** → сразу `moderation_status = APPROVED`.
- Иначе (обычный пользователь или COMMUNITY) → `PENDING`.

В **ленту** попадают только события с `moderation_status = 'APPROVED'` (фильтр в `GET /api/events`).

Админ в панели модерации может: одобрить, отправить на правки (`NEEDS_EDIT`), отклонить (`REJECTED`). Организатору уходит **уведомление** (`server/src/utils/notifications.js`).

### 8.4. Адрес и карта

- `address_public = true` — адрес и метка видны всем в ленте.
- `false` — адрес скрыт до **одобренной** заявки на участие (логика `canViewEventLocation` в `utils.js` на клиенте и проверки на сервере).

### 8.5. Миграции

Отдельного инструмента миграций (Flyway/Liquibase) нет. Схема **идемпотентная**: `CREATE TABLE IF NOT EXISTS`, `ALTER ... ADD COLUMN IF NOT EXISTS` в `db/schema.sql`. При каждом старте Docker API выполняет `schema.sql` и `seed.sql`.

## 9. Аутентификация и роли

### 9.1. Регистрация

1. Клиент запрашивает капчу: `GET /api/auth/captcha`.
2. Пользователь решает пример; ответ проверяется в памяти сервера (`captcha-store.js`).
3. `POST /api/auth/register` — Zod + проверка уникальности email/login.
4. Пароль хешируется: `bcrypt.hash(password, 10)`.
5. Если email в списке `ADMIN_EMAILS` из `.env` → роль `ADMIN`.
6. Возвращаются **JWT** и объект `user`.

### 9.2. Вход

`POST /api/auth/login` — поиск по login, `bcrypt.compare`, выдача JWT.

### 9.3. JWT

В токене payload: `sub` (id пользователя), `role`, `email`, `login`. Срок жизни — **1 день**.

Middleware `requireAuth` извлекает токен из заголовка, кладёт decoded в `req.user`.

`requireAdmin` — дополнительно проверяет `role === 'ADMIN'`.

`optionalAuth` — если токен есть и валиден, заполняет `req.user`; иначе гость.

### 9.4. Три «роли» с точки зрения UI

| Роль | Возможности |
|------|-------------|
| **Гость** | Лента, просмотр, избранное в localStorage |
| **USER** | + создание событий, запись, профиль, отзывы, жалобы |
| **ADMIN** | + модерация, статистика, жалобы, официальные события без очереди (если сам создаёт OFFICIAL) |

## 10. Основные сценарии пользователя

### 10.1. Просмотр ленты

1. `GET /api/events?page=1&limit=10` + фильтры (`q`, `categoryId`, `type`, даты).
2. Сервер: `WHERE moderation_status = 'APPROVED'` + условия фильтров.
3. Клиент рисует карточки и маркеры на карте (только события с видимыми координатами).

Поиск по названию: `ILIKE` или полнотекстовый индекс GIN на `title`.

### 10.2. Запись на мероприятие

1. Пользователь нажимает «Записаться» → модалка с сообщением.
2. `POST /api/events/:id/attend` — создаётся строка в `event_registrations` со статусом `PENDING` (или логика одобрения зависит от настроек организатора).
3. Организатор в «Моё → Я организую» видит участников через `GET /api/events/:id/participants`.
4. `PATCH` заявки — одобрить/отклонить; при лимите мест сервер проверяет, не переполнено ли.

### 10.3. Создание события

1. Форма в `EventFormSection`: поля + выбор точки на карте (`latitude`/`longitude`).
2. Данные организатора подставляются из профиля (`getOrganizerFromUser`).
3. `POST /api/events` → событие в БД со статусом модерации.
4. После одобрения появляется в ленте.

### 10.4. Отзыв

После **окончания** события пользователь с одобренной записью может оставить отзыв: `POST /api/reviews` (проверка на сервере). Один отзыв на пользователя на событие (`UNIQUE`).

### 10.5. Жалоба

`POST /api/events/:id/report` — только на **APPROVED** события. Админ видит список: `GET /api/admin/reports`.

### 10.6. Уведомления

`GET /api/notifications` — колокольчик в шапке. Создаются при модерации, заявках и т.д. на сервере.

## 12. Безопасность

Что можно сказать на защите:

1. **Пароли** — только хеш в БД (bcrypt).
2. **JWT** — секрет в `JWT_SECRET`, не в коде.
3. **SQL** — параметризованные запросы (`$1`, `$2`).
4. **Валидация** — Zod на сервере; не доверяем клиенту.
5. **Права** — проверка владельца события / роли ADMIN в handlers.
6. **Капча** — защита от простых ботов при регистрации.
7. **CORS** — в dev; в Docker фронт и API с одного origin через nginx.

Что **не** реализовано (честно, если спросят): refresh-токены, OAuth, rate limiting, HTTPS в локальном Docker (для прода нужен reverse proxy с TLS).

## 15. Вопросы преподавателя и ответы

### Общие

**В: Что вы разрабатывали?**  
О: Веб-сервис агрегации городских мероприятий с лентой, картой, записью участников, модерацией и отзывами.

**В: Какая архитектура?**  
О: Клиент–сервер, REST API, трёхзвенная: React → Express → PostgreSQL.

**В: Почему PostgreSQL?**  
О: Надёжная реляционная СУБД, удобные типы ENUM, UUID, внешние ключи, подходит для связанных сущностей (пользователи, события, записи).

**В: Почему React?**  
О: Компонентный UI, большая экосистема, удобно делать интерактивную ленту и формы.

**В: Почему не PHP/Laravel / Django / Spring?**  
О: Выбран стек JavaScript на клиенте и сервере (единый язык), Express — лёгкий фреймворк для REST API, подходит по объёму курсовой.

### Фронтенд

**В: Как устроена маршрутизация?**  
О: Hash-роутинг через `window.location.hash` и состояние `page` в `App.jsx`, без React Router.

**В: Откуда данные на экран?**  
О: `useEffect` вызывает функции из `api.js` (fetch), результат в `useState`, React перерисовывает компоненты.

**В: Как работает карта?**  
О: Библиотека Leaflet, тайлы OpenStreetMap, координаты из API; маркеры для событий с известными lat/lng.

**В: Тёмная тема?**  
О: Класс `dark` на `<html>`, стили Tailwind `dark:...`; для авторизованных сохраняется в поле `users.theme` через `PATCH /api/me/theme`.

### Бэкенд

**В: Что такое middleware?**  
О: Промежуточные функции Express. Например, `requireAuth` проверяет JWT до основного обработчика; `validate` — Zod-схему.

**В: Где бизнес-логика?**  
О: В обработчиках маршрутов `server/src/routes/*.js` и утилитах `server/src/utils/`.

**В: Как защищены пароли?**  
О: bcrypt — односторонний хеш; при входе сравниваем хеш, пароль в БД не храним.

**В: Что такое JWT?**  
О: Подписанный токен с данными пользователя; клиент отправляет в заголовке; сервер проверяет подпись секретом.

### База данных

**В: Основные сущности?**  
О: Пользователь, категория, событие, регистрация на событие, избранное, отзыв, уведомление, жалоба.

**В: Связи?**  
О: Событие ссылается на категорию и создателя; регистрации — many-to-many user–event с атрибутами статуса и сообщения.

**В: Как событие попадает в ленту?**  
О: Только при `moderation_status = APPROVED`; community от пользователей сначала PENDING.

### Функционал

**В: Чем OFFICIAL отличается от COMMUNITY?**  
О: OFFICIAL — официальные; если создаёт админ, сразу одобрено. COMMUNITY — от жителей, нужна модерация.

**В: Зачем модерация?**  
О: Чтобы отсеять спам и некорректные публикации до показа всем.

**В: Как работает лимит мест?**  
О: Поле `max_participants`; при одобрении заявки сервер считает APPROVED записи и не даёт превысить лимит.

**В: Избранное у гостя?**  
О: В localStorage; после входа синхронизируется с таблицей `favorites` на сервере.

### Docker и развёртывание

**В: Зачем Docker?**  
О: Воспроизводимое окружение на любой машине: БД + API + фронт одной командой для демонстрации.

**В: Зачем nginx?**  
О: Раздаёт собранный React и проксирует `/api` на backend — один порт 8080 для браузера.

**В: Как инициализируется БД?**  
О: При старте контейнера api выполняется `schema.sql` и `seed.sql` в entrypoint.

### Если спросят сложнее

**В: Почему нет ORM?**  
О: Прямой SQL проще для учебного проекта и прозрачен на защите; пул `pg` + параметры.

**В: Масштабирование?**  
О: Сейчас монолит; горизонтально можно несколько инстансов API за балансировщиком, общая БД, сессии не на сервере (JWT stateless).

**В: SSR?**  
О: Нет, SPA; SEO слабее, для курсовой ленты событий достаточно клиентского рендеринга.

## 16. Словарь терминов

| Термин | Простое объяснение |
|--------|---------------------|
| **API** | Набор URL, к которым клиент обращается за данными |
| **REST** | Стиль API: GET читать, POST создать, PATCH изменить, DELETE удалить |
| **JSON** | Формат обмена данными `{ "key": "value" }` |
| **SPA** | Одностраничное приложение, без перезагрузки всей страницы |
| **Компонент** | Кусок интерфейса React (кнопка, форма, лента) |
| **State** | Состояние компонента; при изменении — перерисовка |
| **Hook** | `useState`, `useEffect` — функции React для состояния и побочных эффектов |
| **Middleware** | Прослойка на сервере до основного кода маршрута |
| **JWT** | Токен авторизации |
| **Хеш пароля** | Необратимое преобразование пароля для хранения |
| **Миграция** | Изменение схемы БД (у нас — через idempotent SQL) |
| **Seed** | Начальные данные (категории) |
| **Прокси** | nginx принимает запрос и пересылает на другой сервис |
| **CORS** | Политика браузера для запросов между разными портами/доменами |
| **Zod** | Библиотека проверки структуры данных |
| **UUID** | Уникальный идентификатор записи |
